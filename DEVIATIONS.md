# DEVIATIONS.md

Log of places where the R2 spec set (Canonical Data Dictionary v0.2 + Core Infra Modules 1-4 + app PRDs) had a genuine gap or conflict with reality, the decision applied, and why. Per the kickoff instructions: real spec gaps get a silent, reasonable resolution + an entry here, not a blocked question.

---

## D0 — Stack: TypeScript modular monolith replacing the existing Python/FastAPI backend

Not a spec gap — an explicit user decision (confirmed via direct question at the start of this rewrite): the existing Python/FastAPI + MedGemma backend (`services/api`, `services/ai`, 1 commit dated 2026-05-11) is retired in favor of the literal R2 architecture (TypeScript, NestJS, Azure OpenAI later). `services/api` and `services/ai` are left in place untouched (still the only working backend for the 3 existing apps) and will be deleted in a dedicated cleanup commit once the new backend reaches functional parity — not deleted now.

Framework/tooling choices made within that agreed architecture (not spec deviations, Module 3 doesn't mandate a specific library):
- **NestJS** (`@nestjs/platform-fastify`) for `core-api` — its modules/guards/interceptors map directly onto Module 3's named domain services + PolicyEngine/AuditWriter shape.
- **`@nestjs/platform-ws`** (raw `ws`), not Socket.IO, for `ws-gateway` — Module 2 §4 specifies a literal custom JSON envelope over `wss://`, not Socket.IO's protocol.
- **BullMQ** for `job-runner`/`ingestion-worker` — Redis-backed, already have Redis in `infra/docker/docker-compose.yml`.
- **node-pg-migrate** (migrations, via `pgm.sql(...)` for reviewable raw SQL) + **Kysely** (typed query builder, no ORM entity mapping) — chosen over Prisma because RLS needs per-request `SET LOCAL` session variables inside a transaction, which Prisma handles poorly in a pooled setup.

## D1 — `packages/domain` instead of overhauling `packages/shared` in place

**Plan said:** overhaul `packages/shared` in place with canonical enums/types.

**Reality found:** the 3 existing apps (patient/family/caregiver) import old pre-R2 types from `@sinalytix/shared` (`TaskType`, `TaskSubtype`, `NotificationType`, `ActorType`, `OnboardingDraftSchema`, etc.) in live screens/stores (e.g. `apps/patient/app/add-task.tsx`, `apps/*/src/store/onboarding.ts`, `apps/*/src/store/notifications.ts`). Rewriting `packages/shared` to match the R2 dictionary would rename/remove values the old apps still reference, breaking `pnpm turbo run lint type-check` for apps explicitly out of scope for this pass (they still run against the old Python backend until their own rewiring phase).

**Decision:** created a new package, `@sinalytix/domain`, holding only R2-canonical types (mirrors the Dictionary exactly, no legacy naming). `packages/shared` is left untouched and keeps serving the 3 existing apps as-is. `packages/domain` is consumed by the new TS backend (`services/core-api` etc.) and will be adopted by each app's own screens/stores as they get rewired to the new API in their respective phase — at that point the corresponding pieces of `packages/shared` are retired in the same commit that rewires that app (not before).

**Why this is safe:** no naming collisions in practice — the old and new enums cover the same domains but with different value sets (e.g. old `NotificationType` has 7 values, new canonical taxonomy v2 has 20+), so keeping them in separate packages prevents an app from accidentally importing a half-migrated type.

## D2 — New backend uses a separate database name (`sinalytix_core`), same Postgres server

**Found:** the legacy Python/Alembic backend already owns tables named `users`, `sessions`, etc. in the `sinalytix` database that `infra/docker/docker-compose.yml` provisions — the exact canonical table names Module 1 requires for the new schema. Migrating into the same database would collide immediately.

**Decision:** the new schema lives in a second database, `sinalytix_core`, on the *same* local Postgres server/container (no docker-compose changes — Postgres trivially hosts multiple databases per instance). `packages/db`'s `db:create` script creates it idempotently if missing. This keeps the legacy backend's data completely untouched during the transition and avoids schema-namespacing workarounds. Production topology (single logical DB, Module 1 §1.1) is unaffected — this is purely a local-dev/transition-period detail resolved once the Python backend is deleted and `sinalytix_core` (or a renamed `sinalytix`) becomes the only database.

## D3 — Faz 0 schema scoping notes (minor, non-blocking)

- **`AuditLogEntry` not partitioned yet.** Module 1 §8.1 specifies monthly partitions; the job that rolls partitions forward (J8, Module 3 §4) doesn't exist until job-runner is populated in a later phase. Built as a plain table now; partitioning added alongside J8 rather than half-built with nothing to maintain it.

## D4 — New backend packages ship compiled `dist/`, not raw `src/*.ts`, as `main`

`packages/shared` (and `ui`/`api-client`) point `main` at `./src/index.ts` because they're only ever consumed by Expo/Metro, which bundles raw TS from `node_modules` directly. `services/core-api` is a plain Node/NestJS process built with real `tsc` (required — NestJS's DI depends on `emitDecoratorMetadata`, which esbuild/tsx/swc don't support, ruling out a tsx-based dev loop). Plain Node can't `require()` a `.ts` file, so `@sinalytix/domain`, `@sinalytix/db`, `@sinalytix/policy-engine`, `@sinalytix/audit` point `main`/`types` at `./dist/index.js`/`.d.ts` instead, and `turbo.json`'s `dev` task now `dependsOn: ["^build"]` so `pnpm turbo run dev --filter=@sinalytix/core-api` builds them first automatically.

**D4b follow-up:** those same 4 packages' tsconfigs originally used `module: "ESNext"` (copied from `packages/shared`'s convention), producing extension-less `export * from './types'` output. Without `"type": "module"` in their `package.json`, Node's loader can't resolve that at runtime (`ERR_MODULE_NOT_FOUND`) — found only once `core-api` was actually booted (`pnpm turbo run dev`), not by the test suite. Since these packages are only ever consumed by `core-api` (CommonJS), switched all four to `module: "CommonJS"` / `moduleResolution: "node"`.

## D5 — Added `oauth_identities` table (not in the Canonical Dictionary)

Find-or-create SSO login (`apple_sso`/`google_sso`, Module 2 §3.1) needs a durable mapping from the provider's verified subject claim to a `User.user_id`. Neither Module 1 nor the Dictionary defines such a table — matching by email alone would be wrong (Apple private-relay emails, email changes over time). Added `oauth_identities` (provider, provider_subject, user_id) as an infra-internal table, the same category Module 3 already uses for `event_outbox`/`job_runs` — required for the system to work, not a canonical domain entity. See `packages/db/migrations/0006_oauth-identities.js`.

## D6 — Signup vs. login vs. OTP division of responsibility (interpretation)

Module 2 §3.1 lists `/auth/signup`, `/auth/otp/request+verify`, and `/auth/login` but only gives one fully worked example (the OTP flow) — it doesn't spell out which auth_methods go through which endpoint, or whether signup/login are find-or-create vs. find-only. Implemented as:
- **`phone_otp`** goes exclusively through `/auth/otp/request` + `/auth/otp/verify` (find-or-create on verify, matching the worked example) — never through `/auth/signup` or `/auth/login`.
- **`apple_sso`/`google_sso`** are find-or-create on *both* `/auth/signup` and `/auth/login` (they're aliases for the same underlying flow) — a mobile SSO button can't know in advance whether the account already exists, so both endpoints must behave the same way for these methods.
- **`email_password`** (HCP only) is the one method with genuine signup-vs-login semantics: `/auth/signup` fails (409) if the email already exists, `/auth/login` fails if it doesn't.
- Initial `User.roles[]` at signup: for `patient`/`family`/`caregiver` app_context, the role is set immediately (`roles = [app_context]`) since they map 1:1. For `hcp` app_context, `roles` stays empty at signup — actual clinical role (clinician/nurse/coordinator/admin) is assigned via org provisioning (C17b), a Faz 5 concern with no Faz 0 endpoint.

## D7 — RLS bypassed by table ownership; added a non-owner `sinalytix_app` role (bug found by the RLS-leak test itself)

**Found while running the Faz 0 RLS cross-org leak test:** it failed — every row was visible with zero session vars set. Root cause: Postgres RLS policies do not apply to a table's **owner** (or superuser) unless the table is explicitly `FORCE ROW LEVEL SECURITY`. Migrations run as `sinalytix` (docker-compose's owner role), so `core-api` connecting as that same role made every policy in the repo silently inert regardless of how carefully written.

**Fix:** `packages/db/migrations/0007_app-role.js` creates a non-owner `sinalytix_app` role with `SELECT/INSERT/UPDATE/DELETE` + `EXECUTE` grants (plus `ALTER DEFAULT PRIVILEGES` so future migrations' tables/functions are covered automatically, no repeated grants per migration). `services/core-api`'s `DbModule` now connects via a new `APP_DATABASE_URL` env var (this role), while `DATABASE_URL` (owner) is reserved for migrations only. Test setup (`test/setup.ts`) exposes both: `ownerDb` for fixture setup/teardown, `appDb` for anything asserting RLS behavior. This isn't a workaround — it's what Module 3 §1.3 already specifies ("api-monolith DB'ye uygulama rolüyle bağlanır... RLS'siz erişim yalnız migration") — the bug was that nothing had made that requirement literally true yet.

**Second-order bug this surfaced:** Postgres re-checks a table's **SELECT** policy for any `RETURNING` clause after an `INSERT` — not just the INSERT policy's `WITH CHECK`. The `users` bootstrap-insert policy (`app_acting_user_id() IS NULL OR user_id = ...`) correctly allows the INSERT itself pre-auth, but `INSERT ... RETURNING user_id` was still rejected, because by the time the RETURNING clause is checked against `users_self_select`, `acting_user_id` still isn't set to match. **Fix applied everywhere a new `User` row is created** (signup, SSO find-or-create, OTP find-or-create, and the equivalent test helper `createUser()` in `test/setup.ts`): generate `user_id` client-side (`randomUUID()`) and `setRlsVar(trx, 'app.acting_user_id', newUserId)` *before* the INSERT, not after. Any future phase creating a new row under a bootstrap-style policy (i.e., a policy with an `IS NULL` escape hatch) and immediately reading it back via `RETURNING` needs the same ordering — this is not specific to `users`, it's a general Postgres RLS + RETURNING interaction worth remembering.

## D8 — Bugs surfaced by updating the branch against `main` (PR #1 review pass)

`faz-0-core-schema-identity` was branched before PR #2/#3/#4 landed on `main` (pnpm version-conflict fix, 318 legacy ruff errors, missing ESLint flat configs for the 3 apps). Merging `main` in to pick up those fixes surfaced three genuine gaps in Faz 0's own work that had never actually been exercised, because CI always died earlier (at the pnpm version-conflict step) before reaching them:

Removed the pin (relies on `packageManager` in `package.json`, same as the frontend job).
2. **`packages/domain`, `packages/db`, `packages/policy-engine`, `packages/audit`, `services/core-api` all had a `"lint": "eslint src/"` script with no `eslint`/`typescript-eslint` devDependency and no `eslint.config.js`.** Invisible before because the TS backend CI job never actually runs `lint` (only `check:rls`, `build`, `type-check`, `test`) — the repo-wide `pnpm turbo run lint type-check` (what the *frontend* job runs, unscoped across the whole workspace) is what catches it, and that job was also dying at the pnpm-conflict step before Faz 0 merged. Added `eslint.config.js` to all 5 (same `tseslint.config(tseslint.configs.recommended, {ignores: ['dist/*']})` pattern PR #3 used for `packages/shared`/`packages/ui`) plus the two devDependencies.

None of these three are `main`'s fault (confirmed by testing `main`'s own tree directly: only its pre-existing `packages/ui` missing-`tsconfig.json` gap reproduces there, unrelated to any of this). ## D9 — Faz 1 Slice 0: dual API clients (`api` + `coreApi`) instead of flipping the base URL wholesale

**Faz 1's implementation plan called for** pointing each app's `src/lib/api.ts` `BASE_URL` at the new `services/core-api` (port 8080, `/v1/*` paths) as part of Slice 0's cross-app plumbing fixes (`X-App-Context` header, `Idempotency-Key`), with each domain's actual endpoint-path/shape rewiring deferred to its own later slice (Slice 2 = EmergencyContact, Slice 3 = PatientFamilyLink, Slice 4 = CaregiverLink).

**Real gap found before writing any code:** every existing call site in all 3 apps (Patient/Family/Caregiver) — tasks, calls, messaging, notifications, health-profile, patient-linking, everything — targets the *old* Python backend's path convention (`/api/v1/...`, port 8000). Simply repointing `BASE_URL` at core-api would 404 every one of those calls immediately, not just the ones a later slice hasn't rewired yet — there is no way to "defer" endpoint-path fixes to their own slice if the shared base URL changes out from under every call at once.

**Fix:** each app's `src/lib/api.ts` now exports **two** clients instead of one:
- `api` — unchanged behavior, still targets the legacy backend (`BASE_URL`, no `X-App-Context`/`Idempotency-Key`, since the old FastAPI backend doesn't read either). Every screen keeps calling this until its own domain slice migrates it.
- `coreApi` — new, targets `services/core-api` (`CORE_API_BASE_URL` = `.../v1`), sends `X-App-Context` + `Idempotency-Key` on mutating calls, and surfaces `Retry-After` via `ApiError.retryAfterSeconds` (Module 2 §6.1/§7.4).

Each subsequent slice's frontend rewire switches its *specific* store's calls from `api` to `coreApi` (and updates the endpoint paths/shapes to match core-api's actual contract) — the two clients coexist for the remainder of the migration; `api` and its legacy-backend target only go away once every screen has been migrated, which is not this phase's job.

**Also fixed in the same pass (pre-existing, unrelated to this migration):** none of the 3 apps' `ApiError`-equivalent types exposed the response body's other fields — `apps/caregiver/app/onboarding/phone.tsx:30` already read `e.body?.detail` off a caught error, relying on an untyped `Object.assign(new Error(...), {status, body})` shape; the new `ApiError` class keeps a typed `body` field so this call site keeps working, verified by grep before assuming it was safe to drop.

**Verified end-to-end** (not just type-checked): started `core-api` locally, replayed the exact request the new `coreApi` client sends (`X-App-Context: patient` + OTP request/verify) against the running server, confirmed a 200 with a real token, then confirmed `X-App-Context` mismatch still correctly 403s. Hit one local-dev-only snag along the way — running the test suite (`pnpm test`) rotates the shared `sinalytix_app` Postgres role's password to match `.env.test` (via `test/setup.ts`'s `ensureAppRolePassword`), which leaves the separately-configured dev `.env`'s `APP_DATABASE_URL` stale until re-synced; not a code bug, just a local multi-environment-sharing-one-role quirk worth remembering next time `pnpm run dev` 500s right after a test run.

## D10 — Faz 1 Slice 1: D7's RLS/RETURNING bug recurs on `consent_grants`; a cross-module NestJS DI gotcha

**D7 bug pattern recurred, found by the SDM-grant-creation test itself, not inferred in advance:** `packages/db/migrations/0009_consent-grant.js`'s `consent_grants_direct_insert` policy correctly lets an active SDM create a grant on a patient's behalf (`WITH CHECK` passes). But the service's `.returningAll()` after that INSERT failed with "new row violates row-level security policy" — Postgres re-checks the table's **SELECT** policy for the implicit `RETURNING` clause, exactly as DEVIATIONS.md D7 already documents for `users`, and `consent_grants_select`'s original two branches (`patient_id = actor` / `granted_to_id = actor`) cover neither side of an SDM-created grant: the SDM is neither the patient nor the grantee. Diagnosed by manually re-testing the policy's boolean expression in isolation (it evaluated `true`) before realizing the *actual* failure was a completely different policy (SELECT, not INSERT) being invoked for `RETURNING`. **Fix:** added a third branch to `consent_grants_select` — an active SDM can also see grants for the patient they represent — which transitively fixes both the INSERT-then-RETURN and UPDATE-then-RETURN (`revoke()`) paths, since both re-check this same SELECT policy. Generalizable lesson beyond this one table: **any INSERT/UPDATE policy with a WITH CHECK/USING branch broader than the SELECT policy's own branches will fail on RETURNING** — when designing a table's RLS from now on, treat "who can write" and "who can read what they just wrote" as one combined design question, not two independently-derived policies.

**Scoped, not fully closed, by design:** `consent_grants_direct_insert` only covers the patient/active-SDM-as-actor paths this slice's own API supports. It deliberately does not yet cover the baseline grant that `PatientFamilyLink` activation will write atomically in Faz 1 Slice 3 (acting user = the family member redeeming a code, who is neither the patient nor an SDM) — that branch needs to reference `patient_family_links`, which doesn't exist until Slice 3's migration. `createBaseline()` (the composable method Slice 3 will call) is built and unit-tested now via the patient-as-actor path (which the current policy already supports), with an explicit code comment flagging that its real calling pattern isn't fully exercisable end-to-end until Slice 3 adds a third, additive INSERT policy branch. This is intentional incremental scoping (Module 3 §9.2's expand-contract philosophy — Postgres OR's multiple policies for the same command together, so Slice 3's addition isn't a breaking rewrite), not an oversight; logged here so it isn't mistaken for one when Slice 3 lands.

**Unrelated NestJS DI gotcha, also found by a test failure (not anticipated):** `AuthContextGuard` (built in Faz 0, used only within `AuthModule` until now) needed to be reused via `@UseGuards(AuthContextGuard)` in the new `ConsentGrantsController`, a different module. Plain `exports: [AuthContextGuard]` from `AuthModule` + `imports: [AuthModule]` in `ConsentGrantsModule` was not sufficient — Nest's guard-resolution mechanism reconstructs `AuthContextGuard` within *each consuming module's own scope* rather than always reusing a single shared singleton, so it also needs `TokenService` (an `AuthContextGuard` constructor dependency) to be resolvable there. Fixed by marking `AuthModule` `@Global()` **and** adding `TokenService` to its `exports` — confirmed by testing incrementally (global alone didn't fix it; exporting `TokenService` too did). Every future domain module needing `AuthContextGuard` should just work now without repeating this — no `imports: [AuthModule]` needed once a module is `@Global()`.

## D11 — Faz 1 Slice 2: `sort_order` uniqueness redesign, a nested-`withRlsContext` stale-read bug, and the onboarding-transfer scope boundary

**`sort_order` uniqueness couldn't be a plain partial unique index — found by the reorder test itself.** The obvious design (`UNIQUE (patient_id, sort_order) WHERE deleted_at IS NULL`) fails a same-statement "swap two contacts' ranks" `UPDATE`: Postgres validates a non-deferrable unique index's collision **per row as each row is touched within a statement**, not once at statement end, so swapping `(A,1)`/`(B,2)` → `(A,2)`/`(B,1)` in one `UPDATE ... FROM (VALUES ...)` trips a spurious duplicate-key error even though the *final* state is unique. A true `DEFERRABLE INITIALLY DEFERRED` constraint genuinely defers checking to commit — but Postgres constraints (unlike indexes) can't carry a partial `WHERE` clause, so the soft-delete exclusion had to move elsewhere. **Fix:** `sort_order` is nullable; soft-delete now sets `sort_order = NULL` (freeing the slot for reuse) alongside `deleted_at`, so a plain non-partial `DEFERRABLE` constraint is sufficient (NULLs never collide). `phone`'s uniqueness has no swap/atomicity need and stays a simple partial index. Surfaced a second, related bug in the same pass: `create()`'s original `sort_order: count + 1` logic assumed slots are always contiguous from the top — removing a non-last contact (e.g. slot 1 of 2) then adding a new one would collide on slot 2 instead of reusing freed slot 1. Fixed by computing the actual lowest free slot from the taken set; regression test added.

**Nested `withRlsContext` silently returns stale data — found by the reorder test after the above fix, no error thrown.** `reorder()`'s original implementation ran its `UPDATE` inside one `withRlsContext(this.db, ...)` transaction, then called `this.list(patientId, actingUserId)` to build the response — but `list()` calls `withRlsContext(this.db, ...)` **again**, which always opens a brand-new `db.transaction()` (a second connection from the pool). Under READ COMMITTED, that second transaction cannot see the first transaction's still-uncommitted `UPDATE`, so it silently returned the pre-swap rows with no error at all — confirmed by directly executing the same raw SQL against the test DB outside the service (the swap itself was correct; only the response-building step was wrong). **Fix:** `reorder()` now builds its response by querying via its own `trx` directly instead of delegating to `list()`. **Generalizable lesson:** any service method that opens its own `withRlsContext`/transaction must never call another method that does the same thing internally, expecting to see its own uncommitted writes — pass the existing `trx` through instead (the same "compose via a shared `trx` parameter" pattern `ConsentGrantsService.createBaseline()` already uses).

**Onboarding's emergency-contact submission intentionally stays on the legacy backend, not rewired this slice.** `apps/patient/app/onboarding/otp.tsx`'s `submitDraft()` posts a single composite payload (`language` + `consent` + `emergency_contact` + `health_seed`) to the legacy backend's `/api/v1/auth/complete-onboarding` in one call — a different domain shape than the plan's Slice 2 scope (`apps/patient/src/store/privacy.ts` + `app/settings/privacy.tsx`, the settings-screen EC management UI). Splitting that composite endpoint apart now would require also re-homing `consent` (already superseded by Slice 1's `ConsentGrant` model with a different shape) and `health_seed` (a different, not-yet-rewritten Faz-1 domain) in the same change — well beyond this slice. Resolution: the onboarding-time EC (added during signup, before Slice 2's dedicated EC endpoints are reachable in that flow) still round-trips through the legacy backend as it always has; the **settings-screen** EC CRUD — the only EC surface this slice's plan named — is fully live against `core-api`. This is a real, load-bearing scope boundary, not an oversight: flagged here so "onboarding's EC field" is never mistaken for evidence that Slice 2 is incomplete.

**Four bugs found by an independent adversarial PR review before merge, all fixed in the same PR:**
1. `reorder()`'s ownership/completeness check (`orderedIds.length === ownedIds.size` + membership) was a count+membership check, not a distinctness check — a duplicate id (e.g. `[A, A]`) passed it while silently omitting another owned contact from the reorder entirely (reproduced live: the omitted contact's `sort_order` was left untouched, no error). Fixed in two places: a `.refine` on `ReorderEmergencyContactsRequestSchema` (packages/domain) rejecting duplicates at the DTO layer (returns a clean 422), **and** an explicit distinctness check inside `reorder()` itself, since the existing unit tests call the service directly and bypass the DTO — belt-and-suspenders given a future caller (e.g. an internal service composing this like `ConsentGrantsService.createBaseline()`) could otherwise reach the service without going through Zod at all.
2. `confirmPhoneVerification()` checked the Redis lockout/fail-count *before* checking whether the caller actually owns the `ec_id` — unlike its sibling `requestPhoneVerification()`, which checks ownership first. This let anyone who merely learns an arbitrary `ec_id` (not otherwise visible to them) submit wrong codes against it, polluting/exhausting that ec_id's rate-limit bucket and locking the real owner out for 15 minutes, without RLS ever being consulted (RLS still prevented any actual data access — this was a rate-limit-bucket-pollution/DoS gap, not a data breach). Fixed by moving an RLS-gated ownership check to the top of the method, mirroring `requestPhoneVerification()`'s existing order.
3. `update()` had no duplicate-phone pre-check (unlike `create()`), so editing a contact's phone to match another active contact's phone fell through to a raw Postgres unique-constraint error (uncaught 500) instead of the 409 the identical logical conflict gets from `create()`. Fixed by adding the same pre-check, scoped to exclude the contact's own row.
4. Frontend: `apps/patient/src/store/privacy.ts`'s `requestPhoneVerification`/`confirmPhoneVerification` wrote `verifyError`/`verifyingEcId` unconditionally in their `catch`/`then` handlers — if a user switched from contact A's verify modal to contact B's before A's in-flight request settled, a late response for A could overwrite B's modal state (wrong error shown, or B's still-open modal force-closed by A's late success). Fixed by guarding each such write with `get().verifyingEcId === ecId`, a no-op if the user has since moved on.

Regression tests added for findings 1-3 in `emergency-contacts.test.ts`; finding 4 has no equivalent RN-component test infra in this repo yet, so it's covered by the store-level guard alone.

## D12 — Faz 1 Slice 3: three Postgres RLS surprises on the family-link fabric, plus family-app auth rewire and scope calls

**The redeem write can't be a plain RLS-governed UPDATE — a sharper Postgres RLS rule than D7/D10, found empirically.** A family member redeeming a `family_link_codes` row is a cross-actor write (they don't own the row; the patient does), so the instinct was D7/D10's pattern: add an explicit `FOR UPDATE` policy letting a non-owner flip `active`→`redeemed`. That silently affected **zero rows** even with the UPDATE policy's `USING` objectively true. Root cause (confirmed by dropping the owner policy and testing the UPDATE policy in isolation against the live DB): **an UPDATE's row-candidate scan requires the OLD row to satisfy an applicable SELECT policy too, not just the UPDATE policy** — UPDATE needs read access to *find* the row, the same way an implicit sub-SELECT does. Granting a SELECT policy wide enough to make an active-but-unredeemed code visible would also expose every active code's plaintext `code`/`qr_payload` to any authenticated user (Postgres RLS has no column-level granularity), defeating the anti-enumeration design. **Resolution:** both the lookup *and* the redeem write go through `SECURITY DEFINER` functions (`family_link_code_lookup`, `family_link_code_redeem` in migration 0011), which bypass the SELECT requirement entirely — same tool and reasoning as 0003's pre-auth lookups. The table needs **no** `FOR UPDATE` or non-owner `FOR SELECT` policy for the redeem itself.

**Same rule bit a second time, one step later.** After redeeming, the service looked up the invited EmergencyContact's `relationship` to copy onto the link — but at that point in the transaction the `patient_family_links` row doesn't exist yet, so the family actor has no RLS path to read that EC row (`emergency_contacts_family_select` requires an *already-active* link). **Resolution:** folded the relationship read into `family_link_code_redeem`'s own `RETURNING` as a scalar subquery, so it's covered by the same SECURITY DEFINER bypass, rather than a separate post-redeem SELECT that RLS would drop to null.

**But a cross-table `EXISTS` proof is still a plain SELECT under RLS — so one narrow non-owner SELECT policy *is* required.** `patient_family_links`'s INSERT policy proves "this family member really just redeemed a code for this patient" via `EXISTS (SELECT 1 FROM family_link_codes WHERE redeemed_by = actor AND status='redeemed')`. That subquery runs under the family actor's own RLS context (not SECURITY DEFINER), so it needs `family_link_codes_redeemer_select` (scoped to `redeemed_by = actor`, which never exposes an unredeemed row). Removing it while refactoring the write to SECURITY DEFINER made the *next* table's INSERT fail — a reminder that moving one statement out of RLS's reach doesn't move the statements that depend on it.

**Two design gaps caught before they became bugs (not RLS-surprises, just foresight):** (1) `ConsentGrantsService.revoke()` opens its own `withRlsContext`, so the unlink flow calling it to cascade the baseline-grant revoke would have hit the exact nested-transaction stale-read bug from D11 — added a `trx`-composable `revokeWithTrx()` instead. (2) `patient_profiles` had only a patient-self SELECT policy, so `listMyLinks()`'s join to render patient display names would have silently returned zero rows for a family member — added `patient_profiles_family_select` (same additive shape as `emergency_contacts_family_select`).

**Frontend scope calls (Slice 3 pairs backend with the app screens that exercise it, per the phase's mandate):**
- **Patient side** (`settings/privacy.tsx`): an "Aileye Davet Et" chip on each still-`pending` emergency contact calls `POST /emergency-contacts/:ecId/invite` and shows the 6-digit code. This is the ec_invite entry point — chosen for testing over a generic connect-code + patient-confirm flow because ec_invite activates instantly (no separate patient-confirm UI needed), giving a self-contained cross-app demo.
- **Family side** — bigger than "two screens", because the family app had the same legacy-auth problem the patient app did (its OTP screens hit the legacy backend, and its own route-drift bug `/family/auth/otp/send` from PR #6). Wired `onboarding/phone.tsx` + `otp.tsx` to core-api `/auth/otp/request`+`/verify` (`app_context:'family'`); `onboarding/connect.tsx` to `POST /family-links/redeem` (handling both the `active` and `pending_patient_confirm` responses — the latter shows a real "waiting for patient" screen instead of assuming success); `store/patients.ts`'s `fetchPatients` to `GET /family/my-links`; and `store/auth.ts`'s `fetchProfile` from the broken legacy `/family/profile` to core-api `/me`. **Deliberately left on legacy / out of scope this slice:** the family app's post-connect main screens (tasks/messages/SOS/reports — those are Faz 3/4), Apple/Google SSO, refresh-token handling in the family auth client (it stores only an access token; a 15-min session is fine for testing, real refresh is a follow-up), and the deep-link "Flow A" auto-connect (its `pending_patient_id` carries a bare patient UUID that the redeem endpoint — which needs a code/qr_payload — can't consume; the deep-link generation that would set it doesn't exist yet, so Flow A stays disabled rather than silently broken). `relationship` on the family side is now per-link (a family member can relate differently to each patient), so it lives on the `/family/my-links` roster, not on a single profile — `fetchProfile` maps it to a placeholder.
- **Verified end-to-end** against the live dev core-api by replaying the exact request shapes both apps now send (patient signup → create EC → invite → family signup → redeem → my-links): the full ec_invite chain returns `status: active` and the family roster shows the patient. This is the concrete "verified end-to-end" for this slice given no simulator.

**Critical finding from the independent review, fixed before merge (C1):** the first cut of this slice authorized redeem's emergency-contact accept-write with a cross-actor UPDATE *policy* (`emergency_contacts_family_invite_accept`). Because Postgres RLS has no column-level `WITH CHECK`, that policy — wide enough to allow the narrow 3-column accept-write — also unlocked the table's pre-existing general-purpose `PATCH /emergency-contacts/:ecId` edit path *and* soft-delete for an active `ec_invite` family member, who could then rewrite the patient's EC phone/name (a cross-tenant write to safety-critical data — that phone is who the SOS chain dials) or delete the contact outright. Reproduced by the reviewer against the live RLS-governed connection. **Fix:** removed the broad UPDATE policy entirely (family members now have *no* UPDATE policy on `emergency_contacts`, only the intended `_family_select` read); the accept-write goes through a new `SECURITY DEFINER` function `family_link_accept_ec(ec_id, family_user_id)` (migration 0012) called only inside the redeem transaction with server-derived args — same tool/reasoning as 0011's redeem functions. Added defense-in-depth owner checks (`patient_id === actingUserId`) to `EmergencyContactsService.update()` and `remove()` so those endpoints are correct regardless of any future RLS policy, not reliant on the absence of one. Regression test added (`family-links.test.ts`): an active ec_invite family member gets 404 on both update and remove, and the contact is unchanged. **Generalizable lesson:** an RLS UPDATE policy authorizes a *whole-row* write for *every* code path that can reach the table, not just the one narrow write you wrote it for — when a cross-actor write touches only specific columns, prefer a SECURITY DEFINER function scoped to exactly those columns over a policy, and never let a mutation endpoint gate solely on RLS visibility when a broader-than-owner SELECT policy exists on the same table.

## D13 — Faz 1 Slice 4: CaregiverLink — a simpler single-table link, the shared redeem rate-limiter, and applying Slice 3's C1 lesson up front

**One table, not two (unlike the family-link fabric).** `caregiver_links` (migration 0013) holds the code *and* the link in one row — a `pending` code becomes a `linked` row on redeem, mirroring the legacy `caregiver_links` model. The family split into `family_link_codes` + `patient_family_links` because it has a `pending_patient_confirm` intermediate state, baseline grants, and an `ec_invite` source. Caregiver linking has none of those (redeem links immediately, no patient-confirm step), so a second table would be ceremony. Legacy rules preserved (LEGACY_HARVEST.md): 6-char uppercase code (the server `upper()`s at lookup, so a lowercased code still resolves — legacy parity), 15-min TTL, status `pending`→`linked` (or `expired`/`unlinked`), `unlinked_by` records `patient|caregiver|system`, generic anti-enumeration 404 on any lookup failure.

**No baseline ConsentGrant on redeem — a deliberate scope boundary.** The family ec_invite path atomically writes a baseline `demographic` grant; the caregiver link writes none. Caregiver clinical-data access is a later CareTeam-phase concern (a caregiver on a shift sees task/med data through a different, shift-scoped mechanism, not a standing personal grant), not Faz 1. A test asserts redeem creates zero `consent_grants` rows so this can't silently regress.

**Slice 3's C1 lesson applied from the start, not retrofitted.** Both cross-actor writes — redeem (caregiver stamps themselves onto a patient-owned row) and unlink (either party ends the link) — go through `SECURITY DEFINER` functions (`caregiver_link_redeem`, `caregiver_link_unlink`), NOT cross-actor RLS UPDATE policies. `caregiver_links` therefore has no caregiver UPDATE policy at all — only the patient-owner `FOR ALL` and a narrow caregiver `FOR SELECT` (scoped `caregiver_id = actor`, so an unredeemed code's plaintext is never exposed). `caregiver_link_unlink` self-authorizes by requiring the actor to be one of the two parties and stamps `unlinked_by` accordingly — so there's no way for a third party to end someone else's link, and no broad UPDATE capability to leak into a general edit path (the exact failure mode C1 was). `patient_profiles` gets an additive `_caregiver_select` policy (active-link-gated, display name only — no clinical data) for the roster/success-screen join, mirroring `_family_select`.

**Shared redeem rate-limiter extracted (the planned "reuse, don't reinvent" step).** Slice 3's inline 5-fail→15-min limiter moved to `common/redeem-rate-limiter.service.ts` (`RedeemRateLimiter`), namespaced per domain (`family-link` vs `caregiver-link`) so the two redeem flows have isolated Redis buckets — burning caregiver-code guesses can't lock you out of family-code redeems. `FamilyLinksService` was refactored onto it (its 97 tests still green), and `CaregiverLinksService` uses the same instance. Both flows now share identical anti-enumeration + lockout behavior by construction.

**Frontend (both consuming apps):**
- **Patient** `settings/privacy.ts`: `generateCaregiverCode` → core-api `POST /patients/:id/caregiver-links` (the code the caregiver redeems). The caregiver-link *status fetch* and *patient-side unlink* stay on the legacy client this slice (they degrade gracefully — legacy down → no linked-state shown → the generate button is what's exercised); rewiring them needs a patient-facing "get my caregiver link" GET that isn't in scope here.
- **Caregiver app** rewired onto core-api (it had the same legacy-auth problem the other apps did): OTP auth (`phone.tsx`/`otp.tsx`, `app_context:'caregiver'`); the onboarding name-transfer moved from the legacy `/caregiver/onboarding/complete` to core-api `PATCH /me` (sets the caregiver's own name on the new backend); `store/patients.ts` `linkPatient` → `POST /caregiver-links/redeem` (**also fixes the pre-existing legacy bug** where the frontend sent a JSON body but the endpoint expected a query param) and returns the linked patient so LINK_01 can confirm success by name; `fetchPatients` → `GET /caregiver/my-patients`; `store/auth.ts` `fetchProfile` → `/me`. `done.tsx` (LINK_01) now shows a success confirmation with the patient's name and a real 429-cooldown / 409-already-linked message. **QR-scan tab deferred:** the caregiver app has no camera dependency and the LINK_01 code path is the testable one; adding `expo-camera` for a QR tab is a follow-up, not worth a new native dep this slice (logged so it isn't mistaken for an oversight — the backend already accepts `qr_payload`).
- **Verified end-to-end** against the live dev core-api: patient signup+name → generate code → caregiver signup → redeem (lowercased, to prove server-side uppercasing) → roster shows the patient by name → unlink → roster empties. All correct.

## D14 — Faz 1 Slice 5 (final): the ApprovalRequest gate — deferred cross-actor execution, a denormalized requester name, and the family_link_permission_change scope call

**The gate re-executes a stored action under a DIFFERENT actor than the requester — and that composes precisely because the action executors are already SECURITY DEFINER.** The flagship gated action is a CAREGIVER-initiated caregiver-link unlink: when the patient has configured `caregiver_link_change` to require approval and has ≥1 eligible family approver, the unlink becomes a `pending` ApprovalRequest instead of taking effect. When a FAMILY member later approves, the unlink must actually happen — but the approve() call runs as the family member, who is neither party to the caregiver link. This works because `caregiver_link_unlink(link_id, actor)` (Slice 4) is SECURITY DEFINER and authorizes by its `actor` param, not the session user: the approve() flow re-runs it with the caregiver id stored in `action_payload`. So the whole "someone approves an action on behalf of the original requester" model falls out of the D12/D13 "cross-actor writes are SECURITY DEFINER functions" discipline for free — no new privilege path. Verified end-to-end live (deferred → family-approved → executed).

**The gate primitive (`ApprovalGateService.maybeGate`) is the reusable choke point.** Called from inside a domain action's own transaction, as the requester: reads the patient's config (SECURITY DEFINER — the caregiver requester can't RLS-read the patient's config), counts eligible approvers (active family members minus the requester, SECURITY DEFINER), then either (a) no config / `requires_approval=false` → run now, no row; (b) required but zero eligible approvers → run now + record `auto_approved_no_approver` (better than blocking forever); (c) required with approvers → create `pending`, do NOT run. The action's own `executeAction` closure is what runs in (a)/(b); (c) defers it to `approve()`. All the cross-actor reads/writes (config read, approver count, request create, decide) go through the migration-0014 SECURITY DEFINER functions; `approval_requests` carries only a SELECT policy (patient + requester + active family members) and NO UPDATE/INSERT policy — same lesson as D12/D13.

**`requested_by_name` is denormalized onto the row (snapshot at creation).** The approvals screen renders the requester's name to the patient/family approver, who have no RLS access to the requester's (e.g. caregiver's) profile — so a live join returned null and the name fell back to a role label (found by the slice's own test). Fix: the gate reads the requester's name from their OWN profile (each profile table is self-readable) at creation and stores it on the request, same point-in-time-provenance reasoning as `audit_log_entries`. This also simplified `listForPatient` to a single-table select. A consequence worth noting: `visibility` and `decide-eligibility` are the SAME "active family member" condition, so there is no "visible-but-ineligible third party" — a non-linked user gets a 404 (never learns the request exists), and the only 403 is the requester-can't-approve-their-own case.

**`family_link_permission_change` is a configurable action type but has NO trigger endpoint this slice — a deliberate scope call.** Its product semantics aren't pinned down in the spec: unlike the clean caregiver case (an outside party's action, overseen by family), "changing a family link's permission_level" has no unambiguous initiator/approver story (a patient changing their own data-sharing shouldn't need family approval; a family member self-raising their own permission is a questionable default). Rather than invent a speculative self-permission-grant flow the product might reject, this slice ships the generic gate + the one semantically-clean, fully-wired, end-to-end-tested action (`caregiver_link_change`), and leaves `family_link_permission_change` valid in the config/enum (a patient can toggle it) with its trigger endpoint awaiting product clarification — flagged to the user, not silently dropped. The gate and the approve/execute dispatch are already generic, so adding it later is a small, additive change.

**48h expiry set at creation; the 24h reminder + expiry-sweep cron is deferred to the job-runner phase** (same precedent as Faz 0's D3 partitioning deferral and Slice 4's shift-24h-alert). `expires_at` is enforced at decide time (an expired pending request can't be approved — the SECURITY DEFINER decide's `expires_at > now()` guard + a service pre-check returning 409), so correctness doesn't depend on a sweep; the sweep only flips stale rows to `expired` for display.

**Frontend — the family approvals screen** (`apps/family/app/approvals.tsx`, the slice's named deliverable): rewired from the legacy `GET /family/patients/:id/approvals` + `PATCH .../:approvalId {status}` to core-api `GET /patients/:id/approvals` + `POST /approvals/:id/{approve,reject}`; renders the server-computed `description` + `requested_by_name`; adds the two previously-missing status branches (`expired`, `auto_approved_no_approver`); and replaces the silent `catch {}` with real 403/409 error surfacing + a re-fetch (approving carries out a server-side action, so the authoritative post-decision state comes from a reload, not an optimistic flip). The patient-config UI and the caregiver-side unlink-trigger UI are API-level this slice (exercised via the e2e), not new app screens — the approvals screen is the plan's named frontend surface.

### D15.1 — Slice 1 (K9/K10) uygulanırken bulunanlar

Migration 0015 + `packages/config` yazılırken planlanmamış dört şey çıktı; ikisi kapatılan güvenlik açığı, ikisi kendi kodumun hatası.

**İki privilege-escalation yolu kapatıldı (planda yoktu, `admin_users`'a dokunurken görüldü).**
1. `admin_users` politikası 0002'de `FOR ALL USING/WITH CHECK (user_id = app_acting_user_id())` idi — yani **herhangi bir kimliği doğrulanmış kullanıcı kendini `admin_users`'a yazabiliyordu**. Bugün etkisiz, çünkü yetki aslında `users.roles` → `app.acting_roles` üzerinden akıyor ve `admin_users`'ı okuyan hiçbir kod yok; ama admin yüzeyi bu tabloyu okumaya başladığı anda doğrudan self-elevation primitifi olurdu. Politika ikiye ayrıldı: self-SELECT + admin-only yazma.
2. Bir seviye yukarısında aynı sınıf: `users_self_update` (0002) yalnız `USING` taşıyor, `WITH CHECK` yok — Postgres bu durumda `USING`'i `WITH CHECK` olarak da kullanır, dolayısıyla kullanıcı kendi satırını **`roles` ve `status` dahil** güncelleyebiliyordu. HTTP yüzeyi bu kolonları açmıyor (PATCH /me yalnız profil tablolarına yazar), ama "bugün kimse çağırmıyor" tam olarak D12'nin cezalandırdığı varsayım. RLS'in kolon-düzeyi `WITH CHECK`'i olmadığı için doğru araç trigger: `reject_self_privilege_escalation()` — `roles`/`status` yalnız admin rolündeki bir aktör tarafından (veya acting user hiç yokken, yani signup bootstrap'ında) değişebilir. Dört regresyon testi yazıldı.

**Kendi migration'ımdaki iki hata, kendi testleri tarafından yakalandı.**
3. `CHECK (array_length(admin_role, 1) >= 1)` boş diziyi **reddetmiyordu**: `array_length('{}', 1)` NULL döner, `NULL >= 1` NULL'dır ve CHECK NULL'ı "ihlal yok" sayar. `COALESCE(..., 0) >= 1` oldu.
4. `system_config.updated_by` / `feature_flags.updated_by` başlangıçta `REFERENCES users(user_id)` idi. Test harness'ı `TRUNCATE users ... CASCADE` ile temizlik yapıyor ve **CASCADE, FK ile referans veren tabloları da boşaltır** — yani her `beforeEach` config seed'ini siliyor, her servis sessizce registry default'larına düşüyordu (config testleri "0 satır" görene kadar fark edilmedi). Kolonlar `audit_log_entries` (0005) ile aynı gerekçeyle düz `uuid` yapıldı: config, herhangi bir hesabın yaşam döngüsünden bağımsız referans veridir. `truncateAll`'ın listesine de açıklayıcı not eklendi.

**Ek olarak:** yazdığım test paylaşılan referans veriyi değiştirip geri yüklemiyordu → başarısız bir koşum sonraki koşumları da bozuyordu (sıraya bağımlı kırılganlık, ürün hatası gibi görünüyor). `afterEach` ile tüm registry geri yükleniyor; süit üst üste iki kez temiz koştu.

**Bağımsız inceleme bulgusu — CRITICAL, merge öncesi kapatıldı (2026-07-23):** Bu slice `admin`'i `AppContext` enum'una ekleyip `sessions`/`consent_records` CHECK'lerini genişletti — ama `initialRoles(appContext)` HCP dışında `[appContext]` döndürüyor ve OTP/SSO signup `app_context`'i yalnız `z.nativeEnum(AppContext)` ile doğruluyordu. Sonuç: anonim bir saldırgan `POST /auth/otp/verify {app_context:"admin"}` ile **`roles=['admin']` bir kullanıcı + admin oturumu** üretebiliyordu — self-service superadmin primitifi, ki bu slice'ın kendi `admin_users` düzeltmesini boşa çıkarıyordu. Bugün HTTP'den kullanılamazdı (admin_users/system_config'e yazan uç henüz yok) ama admin JWT/oturum/rol **şimdi** elde edilebiliyordu ve yazma yeteneği Admin API geldiği an canlı olurdu. Düzeltme: `assertSelfServiceContext()` — OTP ve SSO (signup+login) yalnız patient/family/caregiver'a izin veriyor; `admin` hiçbir zaman self-service değil (Admin PRD §1: superadmin + iki-kişi kuralı ile provizyon), `hcp` de OTP/SSO ile değil yalnız email+şifre+MFA ile (Modül 1 §3.5). İki regresyon testi (`auth-flow.test.ts`): admin/hcp OTP ve SSO signup 400 döner, hiç kullanıcı/oturum/admin_users satırı oluşmaz. İncelemenin doğruladığı ikinci nokta (Low, ileriye not): `Role` enum'undaki `admin` (Faz 5 HCP org-admin) ile platform-admin aynı `app_has_role('admin')` token'ını paylaşıyor — Faz 5'ten önce ayrı `platform_admin` rolü değerlendirilmeli.

**Kapsam notu — `SESSION_POLICY` bilerek `SystemConfig` anahtarı DEĞİL.** Admin PRD A7'nin V0 anahtar listesi oturum sürelerini içermiyor; oturum ömrü spec'in sabitlediği bir güvenlik parametresi, ops'un runtime'da ayarladığı bir şey değil (admin idle timeout'unu uzatabilen bir operatör kontrolü sessizce geri alabilirdi). `packages/config/session-policy.ts`'te kod sabiti olarak duruyor — değiştirmek gözden geçirilmiş bir PR gerektiriyor, config düzenlemesi değil. Aynı gerekçeyle OTP TTL'i (5dk, Modül 2 §3.1) da kod sabiti kaldı.

### D15.2 — Slice 2 (hata zarfı) uygulanırken bulunanlar

**Zarf tam geçiş yapıldı, RFC 7807 bırakıldı** (ürün sahibi kararı). Kazanç zarfın şeklinden çok **kanonik `code`**: istemci artık `CONSENT_REQUIRED` üzerinden dallanıyor, "status kodu + dile göre değişen bir cümle" eşleştirmiyor. Kodun HTTP durumu tek bir tablodan (`API_ERROR_STATUS`) geliyor — aynı kod bir yerde 403, başka yerde 404 olarak çıkamaz.

Spec §1.3 kod listesini "alt-küme" diye veriyor; eksik kalanlar EXT olarak işaretlenip eklendi: `CONFLICT` (domain-durum çakışması, optimistic-lock'tan ayrı), `IDEMPOTENCY_KEY_REUSE`, `BAD_REQUEST` (protokol düzeyi — gövde doğrulaması 422), `FEATURE_DISABLED` (Admin PRD §7/4 zaten adını veriyor), `INTERNAL`.

**`packages/i18n` eklendi — çünkü zarfın `message`'ı `Accept-Language`'a bağlı.** Geçiş öncesi ~45 hata mesajının tamamı throw noktasına gömülü Türkçe literaldi: İngilizce veya Fransızca konuşan bir kullanıcı Türkçe hata görüyordu, ve aynı cümle birkaç yerde ayrı ayrı yaşıyordu. Artık `ApiException` bir **mesaj anahtarı** alıyor (kataloğa tiplenmiş — yazım hatası derleme hatası), filtre `Accept-Language`'ı çözüp EN/FR/TR'den metni üretiyor. Katalog testi üç dilin **placeholder kümesinin aynı olduğunu** da doğruluyor: bir çeviri `{minutes}`'i düşürürse yalnız o dilde "birkaç dakika sonra" gibi bozuk bir cümle çıkardı ve gözle görülmezdi.

**Filtrede üç değişmez, her biri ayrı bir sızıntı yolu olduğu için yazılı:** (1) `message` yerelleştirilir, `code` asla; (2) beklenmeyen hatanın kendi metni istemciye **hiç** gitmez (ham exception metni satır/sorgu/PHI taşıyabilir — Modül 2 §8), `request_id` ile sunucuda loglanır; (3) `ApiException` olmayan bir `HttpException` (Nest'in kendi veya bir kütüphanenin fırlattığı) kendi İngilizce metnini iletemez — status'tan kanonik koda eşlenip katalogdan yeniden yazılır.

**B2:** başlık `X-Idempotency-Key` oldu; replay yanıtı `X-Idempotent-Replayed: true` taşıyor (§1.3'ün `IDEMPOTENCY_REPLAY` 200'ü bir hata zarfı değil, orijinal başarı gövdesi). Sunucu çıplak `Idempotency-Key` yazımını **okumada hâlâ kabul ediyor** — kullanıcıların telefonundaki mevcut uygulama derlemesi deploy anında 400 almasın diye; istemciler X- formunu gönderiyor.

**Yol üstünde bulunan gerçek boşluk:** `/auth/otp/request` telefonu `z.string().min(8)` ile doğruluyordu — yani 8 karakterlik herhangi bir dize `phone_e164` kolonuna yazılabilirdi, oysa `EmergencyContact` gerçek E.164 formatını zorluyordu. İki ayrı regex kopyası olması da ikinci bir uyuşmazlık şansıydı. Tek `E164PhoneSchema` (`packages/domain`) yapıldı ve her iki yer ona bağlandı. Zarf testinin kendisi yakaladı.

**Zod mesajları artık istemciye gitmiyor:** doğrulama hatası `details[] = {field, issue}` olarak çıkıyor, `issue` zod'un **kod** slug'ı (`invalid_type`), zod'un insan cümlesi veya **gönderilen değer** değil. Bir telefon numarası PII'dır; `details[]` makine-okunur ve PHI'sız kalmak zorunda (Modül 2 §8). Test bunu reddedilen değerin yanıtta geçmediğini arayarak doğruluyor.

**İstemci tarafı:** üç uygulamanın `ApiError`'ı artık `code` taşıyor (ekranlar buna dallanmalı, `message`'a değil — mesaj yerelleştirilmiş ve değişecek), `Accept-Language` gönderiliyor, ve hata gövdesi ayrıştırması hangi backend olduğuna göre dallanıyor (core-api `error.message` / legacy `detail`) — yanlış olanı okumak sessizce "undefined" üretirdi. Patient app'in Apple/Google SSO ekranları legacy backend'e gittiği için `detail` okumaya devam ediyor, doğrusu bu.

### D15.3 — Slice 3 (`/consents` + declarative_v0)

**B4 — `consent_records` tablosu Faz 0'dan beri var ama hiç ucu yoktu**, yani üründe onam kaydedebilen hiçbir yol yoktu. Bu bir kolaylık eksiği değil, uyum yükü taşıyan bir boşluk: PIPEDA sürümlü, değiştirilemez, açık rıza istiyor (Uyum Listesi §2). `POST /consents` + `GET /consents` eklendi.

> **DÜZELTME (bağımsız inceleme, 2026-07-23) — C17a yolu ilk halde 500 veriyordu, "çalışıyor" iddiam YANLIŞTI.** `create()` `INSERT ... RETURNING` yapıyordu; klinisyen-adına yolda `resolveSubject` `user_id`'yi HASTAYA ayarlıyor, ama 0004'ün SELECT policy'si `user_id = app_acting_user_id()` (klinisyen). Postgres RETURNING'in döndüreceği satıra SELECT policy'sini yeniden uygular ve o satır (hasta-sahipli) policy'yi geçemez → RLS hatası → filtre 500'e eşler, hiçbir onam kaydedilmez. Bu tam olarak D7/D15.4b'nin "RETURNING SELECT policy'sini yeniden kontrol eder" tuzağı — Slice 3 onu sevk etti, Slice 4b aynı sınıfı `ec_change` için SECURITY DEFINER ile çözmüştü. **Düzeltme (migration gerektirmez):** `consent_id` + `server_recorded_at` uygulama tarafında üretiliyor ve INSERT `RETURNING` olmadan koşuyor — dönen satır okuması tamamen atlanıyor. `server_recorded_at` API sunucusunun kendi saati, ki bu zaten alanın anlamı (Modül 1 §4.1); DB default'una göre kayıp yok. C17a başarı yolu artık servis-seviyesinde test ediliyor (klinisyen app'siz hasta adına kaydeder → satır hastaya ait, `clinician_recorded`). İlk yazdığım test yalnız non-klinisyen 403'ünü kapsıyordu (DB'ye hiç ulaşmıyor), başarı yolu test edilmiyordu.

**İki Low (aynı inceleme):** (S3-3) Patient app `submitConsentRecord` `res.ok`'i kontrol etmiyordu — 4xx/5xx başarıdan ayırt edilemiyordu; artık non-2xx'te throw ediyor (çağrı yerleri zaten non-fatal sarmalıyor, ama sessiz "başarı" yok). (S2-1) `mapStatusToCode` sıralanmamış status'lar için `INTERNAL` dönüyor ama orijinal status'u koruyordu — framework-fırlatılan bir 405 → HTTP 405 + `code:INTERNAL` uyuşmazlığı; 405 → `METHOD_NOT_ALLOWED` eklendi.

**Sunucu-türetilen alanlar bilinçli:** `user_id`, `server_recorded_at`, `ip_hash` istemciden alınmaz — bir istemci gövdeye başka bir id koyarak başkası adına onam iddia edemez, ve sunucunun "ne zaman ulaştı" kaydını geriye alamaz. `consented_at` ise **kasıtlı olarak istemciden** gelir: insanın kabul'e bastığı andır ve isteği önceleyebilir (çevrimdışı onboarding).

**`GET /consents/effective` eklendi (spec'te yok, karar).** `ConsentRecord` append-only ve sürümlü; canlı cevap `app_context` başına **en yeni satır**. Spec yalnız yazma + geçmiş ucu tanımlıyor, dolayısıyla her çağıran (Modül 4 §1'in AI onam kapısı, onboarding devam ekranı, ayarlar toggle'ı) "en yeni kazanır"ı kendi türetirdi ve biri yanlış yapardı. B9 bunu somutlaştırıyor: `ack_ai_processing` tek yerde yaşıyor, o hâlde onun tutulup tutulmadığını da tek yer hesaplamalı. **Bayraklar sürümler arasında BİRLEŞTİRİLMEZ** — v1'de AI'yı kabul edip v2'de reddeden bir kullanıcıda birleşim, geri çekilmiş bir onayı diriltirdi. Test tam bu senaryoyu tutuyor.

**`METHOD_NOT_ALLOWED` (405) kodu eklendi.** Spec §3.2 "PATCH/DELETE yok, 405" diyor; Nest'in varsayılanı 404 olurdu. Fark önemli: `PATCH /consents/{id}` üzerine 404, yanlış URL gibi okunur ve istemci yazarını doğrusunu aramaya davet eder; 405 "bu işlem var olamaz" der. DB katmanı zaten trigger ile zorluyor (migration 0004) — bu onun kibar dış katmanı.

**B5 — `enforcement: "declarative_v0"`** artık her grant DTO'sunda. Modül 2 §3.2 bu farkın istemciye **açıkça bildirilmesini** şart koşuyor. Boolean değil literal: tam motor gelince değer `"enforced_v1"` olacak ve buna dallanan istemci sessizce eski semantiği varsaymak yerine gürültüyle kırılacak.

**Ön yüz (kullanıcının "her slice gerçek ön yüzle eşleşsin" kuralı):** Patient app'in onboarding onam draft'ı yalnız legacy `complete-onboarding`'e gidiyordu — EC ve health seed ile aynı best-effort çağrıda. Artık auth'tan hemen sonra kendi çağrısıyla core-api'ye gidiyor (üç giriş yolunun üçünde de: OTP, Apple, Google); legacy çağrı EC + health seed'i kendi slice'larına kadar taşımaya devam ediyor (D9 çift-istemci deseni). `TOS_VERSION` tek sabit — kaydedilen sürüm kullanıcının okuduğu ekranla uyuşmazlık yapamaz.

### D15.4 — Slice 4 (K6 + FAM-12 hizalaması)

**B3 — `PATCH /family-links/{id}` hiç yoktu, dolayısıyla K6'nın sunucuda hiçbir uygulama noktası yoktu.** Canlı bir ihlal üretmiyordu çünkü kabul akışı `permission_level`'ı her zaman `view` yazıyor ve değiştirecek bir yol yok; kural sadece **yoktu**, onu gerektirecek ilk ucu bekliyordu. Uç üç ayrı kapıyla eklendi:
- **Yalnız Patient app** (FAM-13). "Yalnız hasta" değil — `app_context` önemli, çünkü tek bir `User` satırı birden çok rol taşıyabilir (Sözlük §1); aile uygulamasındaki bir oturum kendi erişimini yükseltebilmemeli.
- **Hasta link sahibi.** Sahip değilse **404, 403 değil**: aile üyesi bu satırı okuyabiliyor, dolayısıyla 403 hangi link id'sinin ona ait olduğunu doğrulardı (varlık-sızdırmazlık, Modül 1 §11).
- **K6.** `full` için bu hasta–aile üyesi çifti adına **aktif** `SDMDeclaration` şart. `full`, vekil karar verici ağırlığı taşıyan düzey (onay kuyruğunun yetkili karar verici saydığı şey) — beyansız verilmesi PHIPA açısından anlamlı bir rolü UI toggle'ı ile dağıtmak olurdu. Testler dört reddi ayrı ayrı tutuyor: beyan yok / beyan pasif / beyan **başkasını** gösteriyor / yanlış app_context.

**A5 — aksiyon tipleri FAM-12 §3 + Modül 1 §9'a hizalandı.** `family_link_permission_change` **iki kaynakta da yok**; D14 zaten "ürün semantiği belirlenmemiş, trigger ucu yok" diye işaretlemişti — uydurmaydı, o yüzden ona uç yazmak yerine kaldırıldı. Yerine spec'in dördü: `caregiver_link_change`, `ec_change`, `profile_edit`, `account_delete`.

**FAM-12'yi okurken çıkan iki gerçek davranış sapması (ikisi de düzeltildi):**

1. **Varsayılan terstiydi.** Migration 0014'ün `approval_config_requires_approval` fonksiyonu yapılandırılmamış aksiyon için `false` dönüyordu. "Gated değil" varsayılanı, ayarı hiç açmamış her hasta için — yani nüfusun tamamı, çünkü onları oraya yönlendiren hiçbir şey yok — aile güvenlik ağını sessizce kapatıyordu.
